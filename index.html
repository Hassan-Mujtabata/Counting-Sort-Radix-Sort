Welcome to Non-Comparison Sorting

This tool visualizes Counting Sort and Radix Sort. These algorithms achieve linear sorting time by exploiting the properties of integers, bypassing the mathematical $\Omega(n \log n)$ lower bound.

Worked Example: Counting Sort on [2, 1, 1, 0, 2]

Max value $k = 2$. Size $n = 5$. We initialize count array `C[0..2] = [0, 0, 0]`.

  1. Count Phase: Scan input `A`. Tally occurrences in `C`.
    After counting: `C = [1, 2, 2]` (One 0, Two 1s, Two 2s).
  2. Prefix Sum Phase: Add previous counts to current (`C[i] += C[i-1]`).
    `C[1] = 1+2 = 3`. `C[2] = 3+2 = 5`. Final `C = [1, 3, 5]`.
    Meaning: Elements $\le 1$ will occupy up to position 3.
  3. Placement Phase (Right-to-Left for Stability):
    - Read A[4] = 2. Place at output index `C[2]-1` = 4. `C[2]` becomes 4.
    - Read A[3] = 0. Place at output index `C[0]-1` = 0. `C[0]` becomes 0.
    - Read A[2] = 1. Place at output index `C[1]-1` = 2. `C[1]` becomes 2.
    - Read A[1] = 1. Place at output index `C[1]-1` = 1. `C[1]` becomes 1.
    - Read A[0] = 2. Place at output index `C[2]-1` = 3. `C[2]` becomes 3.
    Final Output: [0, 1, 1, 2, 2]

CLO-1: Complexity Analysis & Formal Asymptotics

Comparison-based sorts are mathematically bounded to $\Omega(n \log n)$. Below is the Decision Tree visualizing why.

Decision Tree Lower Bound: $\Omega(n \log n)$

Sorting 3 elements (a,b,c) requires distinguishing $3! = 6$ possible permutations. The tree height $h$ must have enough leaves to cover $n!$ outcomes. Thus, $2^h \ge n! \implies h = \Omega(n \log n)$.

CLO-3: Design Paradigm Comparison

Why isn't Counting Sort a standard paradigm? Let's compare the structural approach visually.

Divide & Conquer (e.g., Merge Sort)

Recursively splits the array into subproblems, solves them, and merges the results. Heavy call stack usage.

Integer Exploitation (Counting Sort)

Flat, linear passes. No recursion, no splitting. Exploits array indexing in memory directly.

Counter-Example: Why Greedy Fails Here

A Greedy algorithm makes a locally optimal choice at each step. If we tried to sort `[5, 2, 8]` greedily by simply finding the minimum, placing it, and repeating, we degrade exactly back to Selection Sort ($O(n^2)$).

Counting Sort avoids this trap by not making "choices". It uses the global histogram of the data to perform deterministic index arithmetic.
