📚 Educational Content — Worked Example, Key Concepts & Common Mistakes
✏️ Worked Example: Counting Sort on [3, 1, 4, 1, 5]
1
Build count array (k=5):
Scan input → C = [0,2,0,1,1,1]
C[1]=2 because 1 appears twice.
2
Prefix sums (cumulative):
C = [0,2,2,3,4,5]
C[i] now = # elements ≤ i.
3
Place right-to-left (for stability):
A[4]=5 → B[C[5]-1]=B[4], C[5]=4
A[3]=1 → B[C[1]-1]=B[1], C[1]=1
A[2]=4 → B[C[4]-1]=B[3], C[4]=3
A[1]=1 → B[C[1]-1]=B[0], C[1]=0
A[0]=3 → B[C[3]-1]=B[2], C[3]=2
4
Result: B = [1, 1, 3, 4, 5] ✓
Total ops: O(n+k) = O(5+5) = O(10)
⭐ Key Concepts
Stability Matters
Right-to-left placement ensures equal elements keep their original relative order. This is critical for Radix Sort correctness — each digit pass must preserve previous passes.
Integers Only
Counting Sort works by using values as array indices. It cannot handle floating-point numbers, strings, or arbitrary objects. The range k must be known in advance.
When k ≫ n, O(k) Dominates
Sorting [1, 9999] costs O(n+k) = O(10001) — far worse than O(n log n) = O(2). Always check if k is proportional to n before using Counting Sort.
Radix Sort = Repeated Counting Sort
Each digit pass is a full stable Counting Sort on that digit position. Sorting by LSD → MSD ensures the final pass produces a globally sorted array.
⚠️ Common Mistakes
✗
Iterating left-to-right when placing: This breaks stability. Equal elements would be placed in reverse order. Always iterate right-to-left in the output phase.
✗
Assuming O(n+k) is always faster: If k=1,000,000 and n=10, Counting Sort is far slower than Quicksort. Only use it when k = O(n).
✗
Using an unstable sort inside Radix Sort: If the sub-sort is unstable, previous digit passes are destroyed. Counting Sort must be stable here.
✗
Sorting MSD first in Radix Sort: You must sort LSD (least significant digit) first, then work towards MSD. MSD-first requires a more complex recursive approach.
✗
Forgetting C has size k+1: Values range from 0 to k, so the count array needs k+1 slots, not k.

🌳 Decision Tree — Why Comparison Sorts Need Ω(n log n)
Every comparison-based sort corresponds to a decision tree. Each internal node is a comparison A[i]≤A[j]; each leaf is one possible sorted permutation. A tree of depth h has at most 2ʰ leaves, but needs at least n! leaves (one per permutation of n elements). Therefore h ≥ log₂(n!) ≈ n·log n, proving the Ω(n log n) lower bound. Counting & Radix Sort bypass this by never making comparisons — they exploit integer structure instead.
🔴 Worst Case Analysis — When and Why Performance Degrades
Counting Sort Worst Case
When k ≫ n (the value range vastly exceeds the element count), the O(k) initialization and prefix-sum passes dominate. Example: sorting [1, 9999] requires a count array of size 10,000 for just 2 elements — effectively O(k) = O(10000) vs O(n log n) = O(2). The "worst case" preset [all equal values] is a special case: k=value of that element, but since all elements are the same, the placement phase is trivially fast. The true worst case is a small array with one very large value.
!
Try input: 1, 9999 — notice the count array must be 10,000 cells wide for just 2 elements.
Radix Sort Worst Case
When d is large (many digits per number), Radix Sort makes d full passes each costing O(n+b). Example: sorting 8-digit numbers (d=8) costs 8×O(n+10). For numbers up to 10⁹ (d=10 in decimal), this is still very competitive. The practical worst case is when values have many digits AND n is small — each pass has constant overhead.
!
Try Radix Sort on: 100000000, 200000000, 300000000 — watch it take 9 digit passes.

🔍 Role of Sorting Within Each Algorithm
Counting Sort
Counting Sort is the sorting algorithm. The core mechanism is: (1) Use values as array indices to count occurrences — this implicitly sorts because array indices are ordered. (2) Prefix sums convert counts to final positions. (3) Placement delivers each element to its correct sorted index. No element is ever compared to another.
Radix Sort
Radix Sort uses Counting Sort as a stable sub-sort on each digit position. The sorting of individual digits is the engine that drives global sorting. Without a correct stable sort at each pass, the entire algorithm produces wrong output. The highlighted animation in the main visualizer above shows each digit-pass as a distinct Counting Sort run — each one is a complete sort of the current array by one digit position.
📈 Scaling Behaviour — When Each Algorithm Wins
Counting Sort Wins When
• k = O(n) — value range proportional to input size
• Data is integers in a known, bounded range
• Stability is required
• Memory is available (O(n+k) space needed)
Example: sorting exam scores 0–100 for 10,000 students → k=100, n=10000, clearly O(n).
Quicksort / Merge Sort Win When
• Data is not integers (strings, floats, objects)
• k ≫ n (sparse values over a wide range)
• Memory is tight (Quicksort is O(log n) space in-place)
Example: sorting 100 UUID strings — can't use Counting Sort at all.

🧩 CLO-3 — Design Paradigm: Not D&C, Not DP, Not Greedy 10 marks
Which Paradigm Does This Use?
✗ Not Used
Divide & Conquer
D&C splits the problem into smaller sub-problems, solves them recursively, and combines results. Examples: Merge Sort (split array in half, sort each, merge), Quicksort (partition around pivot, recurse).

Counting Sort does not split the array. There is no recursion, no sub-problem structure, no combine step.
✗ Not Used
Dynamic Programming
DP solves overlapping sub-problems and stores results to avoid recomputation. The count array is sometimes confused for a DP table, but it is not — there are no overlapping sub-problems being memoized. Counting Sort computes a frequency distribution in one linear scan, with no recurrence relation.

Examples of actual DP: Longest Common Subsequence, Matrix Chain Multiplication.
✗ Not Used
Greedy
Greedy algorithms make locally optimal choices at each step, hoping to reach a global optimum. Counting Sort makes no "choices" — it deterministically maps every element to its exact output position. There is no decision being made; placement positions are arithmetically computed from prefix sums.

Examples of Greedy: Dijkstra's algorithm, Huffman coding, Activity Selection.
✅ What Paradigm Counting Sort & Radix Sort Actually Use: Integer Structure Exploitation
The Core Insight
Counting and Radix Sort belong to a class of non-comparison-based integer sorting algorithms. They do not fit any of the three standard paradigms (D&C, DP, Greedy) because their design principle is fundamentally different: they exploit the algebraic properties of integers to place elements directly without comparisons.
How They Exploit Integer Structure
• Integers can be used as array indices — this is the key. If A[i]=7, you can directly access C[7]. No floating-point or string can do this.
• Integer digits are extractable — getDigit(n, pos) isolates any digit in O(1). This allows Radix Sort's digit-by-digit passes.
• Integer ranges are bounded and enumerable — you can preallocate exactly C[0..k] and know every possible value in advance.
Why This Beats Ω(n log n)
The Ω(n log n) lower bound applies only to algorithms that determine ordering exclusively through comparisons. By using integer values as indices, Counting Sort never compares A[i] to A[j] — it simply reads A[i] and uses its value to address C[A[i]]. This completely sidesteps the decision-tree argument.
The Algorithm Pattern
The pattern is: Count → Accumulate → Place. This three-phase structure is unique to integer sorting. No D&C recursion, no DP table, no greedy choice — just direct positional arithmetic computed from integer properties.
🌲 D&C Comparison: Merge Sort Recursion Tree (for [4,2,2,8])
Merge Sort divides [4,2,2,8] into halves recursively (orange dashed), then merges sorted sub-arrays (green). This is classic D&C: split → conquer → combine. The recursion tree has O(log n) levels. Counting Sort has no such tree — it never splits the array.
➡️ Counting Sort Flow (for [4, 2, 2, 8]) — Linear, No Recursion
Counting Sort flows in three sequential linear passes. The array is never split. There is no recursion tree. No sub-problem is identified. This is fundamentally not D&C — it's direct positional computation exploiting integer indexing.
❌ Counterexample — Why Other Paradigms Give Worse or Wrong Results Here
If You Tried D&C (like Merge Sort) on Integer Data
Merge Sort works correctly on integers, but pays O(n log n) comparisons when Counting Sort can sort in O(n+k) with no comparisons. For n=1,000,000 elements in range 0–1000, Merge Sort needs ~20,000,000 comparisons, while Counting Sort needs only ~1,001,000 operations. The D&C paradigm ignores the integer structure and pays a higher cost unnecessarily.

If You Tried Greedy (pick the smallest each time)
A naive greedy approach — repeatedly pick the minimum element — is Selection Sort, which runs in O(n²). It makes n passes over the array, each requiring a full scan. This ignores integer structure completely and is far worse than O(n+k).
If You Tried DP (build a sorted subsequence with a table)
Applying DP to sorting (like finding Longest Increasing Subsequence) gives O(n log n) at best — still worse than O(n+k) for suitable k, and doesn't produce a full sort in O(n+k). The DP approach treats elements as opaque objects, while Counting Sort treats their values as addresses.
!
Key takeaway: None of the three classic paradigms can achieve O(n+k) for integer sorting. Only by exploiting integer-specific properties (values as indices, digit extraction) can we break the Ω(n log n) barrier.
