What is Counting Sort & Radix Sort?

A non-comparison sort is an algorithm that sorts integers without ever comparing one element to another. For example, imagine sorting 10,000 exam scores in the range 0–100. Instead of checking every pair, Counting Sort simply counts how many students scored each value — and places them directly. How many steps does that take? Just T(n) = O(n + k) — one pass to count, one pass to place.

Why Do We Care?

These algorithms let us sort without ever comparing elements. By analyzing their complexity, we understand whether an algorithm scales gracefully like O(n), reasonably like O(n log n), or catastrophically like O(2ⁿ).

How to Use This Tool

⭐ Key Concepts
Stability Matters
Right-to-left placement ensures equal elements keep their original relative order. This is critical for Radix Sort correctness — each digit pass must preserve previous passes.
Integers Only
Counting Sort works by using values as array indices. It cannot handle floating-point numbers or strings. The range k must be known in advance.
When k ≫ n, O(k) Dominates
Sorting [1, 9999] costs O(n+k) = O(10001) — far worse than O(n log n) = O(2). Always check if k is proportional to n before using Counting Sort.
Radix Sort = Repeated Counting Sort
Each digit pass is a full stable Counting Sort on that digit position. Sorting LSD → MSD ensures the final pass produces a globally sorted array.

🌳 Decision Tree — Why Comparison Sorts Need Ω(n log n)
Every comparison-based sort corresponds to a decision tree with n! leaves. A tree of depth h has at most 2ʰ leaves, so h ≥ log₂(n!) ≈ n·log n, proving the Ω(n log n) lower bound. Counting & Radix Sort bypass this by never comparing elements.
🔴 Worst Case Analysis — When and Why Performance Degrades
Counting Sort Worst Case
When k ≫ n, the O(k) initialization and prefix-sum passes dominate. Example: sorting [1, 9999] requires a count array of 10,000 cells for just 2 elements — O(k)=O(10000) vs O(n log n)=O(2). The "all equal values" case is actually NOT the worst case — k equals that one value, but placement is trivially fast. The true worst case is a tiny array with one very large value.
!
Try the Worst case preset [1, 9999] — notice the huge count array needed for just 2 elements.
Radix Sort Worst Case
When d is large (many digits per number), Radix Sort makes d full passes each costing O(n+b). For numbers up to 10⁹ (d=10 in decimal), this is still very competitive. The practical worst case is large d with small n — each pass has constant overhead.

📝 Fully Worked Example — Counting Sort on [4, 2, 2, 8]
Phase 1 — Count (O(n) time)
1
Initialize C[0..8] = [0,0,0,0,0,0,0,0,0] — k=8 (max value), so we need 9 cells.
2
Scan A left→right: A[0]=4 → C[4]++, A[1]=2 → C[2]++, A[2]=2 → C[2]++, A[3]=8 → C[8]++.
3
Result: C = [0,0,2,0,1,0,0,0,1]. Index = value, cell = count of occurrences.
Phase 2 — Prefix Sums (O(k) time)
4
For i=1 to 8: C[i] += C[i-1]. After: C = [0,0,2,2,3,3,3,3,4].
5
Now C[i] = number of elements ≤ i. So C[4]=3 means 3 elements are ≤ 4: the two 2's and the 4 itself.
Phase 3 — Place Right-to-Left (O(n) time)
6
Scan A right→left (j=3 downto 0) to preserve stability for equal values.
7
j=3, A[3]=8: B[C[8]−1] = B[3] = 8. Decrement C[8] to 3.
8
j=2, A[2]=2: B[C[2]−1] = B[1] = 2. Decrement C[2] to 1.
9
j=1, A[1]=2: B[C[2]−1] = B[0] = 2. Decrement C[2] to 0.
10
j=0, A[0]=4: B[C[4]−1] = B[2] = 4. Decrement C[4] to 2.
✓
Output: B = [2, 2, 4, 8]. Total ops = n + k + n = 4+8+4 = 16. No comparisons made.
💡 Key Concepts to Remember
Why right-to-left in Phase 3?
Scanning right-to-left ensures equal elements appear in the same relative order in the output as they did in the input — this is what makes Counting Sort stable. Stability is critical when Counting Sort is used as a sub-routine inside Radix Sort.
What does C[i] mean after prefix sums?
After the prefix sum pass, C[i] tells you: "there are C[i] elements with value ≤ i." This directly gives the 1-based output position of the last element with value i. Decrementing after each placement shifts earlier equal elements one spot left.
When is O(n+k) actually O(n)?
When k = O(n) — i.e., the value range grows proportionally with n. Example: sorting n exam scores in [0,100] → k=100 is constant, so it's truly O(n). Sorting n random integers in [0, n²] → k=n² makes it O(n²), worse than merge sort.

🔍 Role of Sorting Within Each Algorithm
Counting Sort
Counting Sort is the sorting algorithm. Values are used as array indices — this implicitly sorts because indices are ordered. Prefix sums convert counts to final positions. No element is ever compared to another. Watch for the orange "Active" bar in the visualizer — it shows the element being placed into its sorted position.
Radix Sort — Counting Sort as Sub-Sort
Radix Sort uses Counting Sort as a stable sub-sort on each digit position. Each pass banner in the visualizer ("PASS N — Counting Sort on [digit] digit") shows exactly when a full Counting Sort run begins. Without a stable sub-sort, previous digit passes are destroyed.

📈 Scaling Behaviour — When Each Algorithm Wins
Counting Sort Wins When
• k = O(n) — value range proportional to input size
• Data is non-negative integers in a known bounded range
• Stability is required and memory is available
Example: sorting exam scores 0–100 for 10,000 students → k=100, clearly O(n).
Quicksort / Merge Sort Win When
• Data is not integers (strings, floats, objects)
• k ≫ n (sparse values over a wide range) — try preset [1, 9999]
• Memory is tight (Quicksort is O(log n) space in-place)
Example: sorting 100 UUID strings — can't use Counting Sort at all.

🧩 CLO-3 — Design Paradigm: Not D&C, Not DP, Not Greedy 10 marks
Which Paradigm Does This Use?
✗ Not Used
Divide & Conquer
D&C splits the problem recursively and combines. Examples: Merge Sort, Quicksort.

Counting Sort does not split the array. There is no recursion, no sub-problem structure, no combine step.
✗ Not Used
Dynamic Programming
DP stores results of overlapping sub-problems. The count array is sometimes confused for a DP table, but it is not — there are no overlapping sub-problems being memoized, no recurrence relation.

Examples of DP: LCS, Matrix Chain Multiplication.
✗ Not Used
Greedy
Greedy makes locally optimal choices. Counting Sort makes no "choices" — placement positions are arithmetically computed from prefix sums. There is no decision being made.

Examples of Greedy: Dijkstra's, Huffman coding.

Phase 1: COUNT — O(n) time
For each A[i], we write C[A[i]]++. The value of A[i] is used directly as an array index. This is purely exploiting the integer property — integers can be array addresses. No comparison, no decision, no sub-problem.
1
Input A = [4, 2, 2, 8] — allocate C[0..8] = all zeros
2
A[0]=4 → C[4]++ → C = [0,0,0,0,1,0,0,0,0]
3
A[1]=2 → C[2]++ → C = [0,0,1,0,1,0,0,0,0]
4
A[2]=2 → C[2]++ → C = [0,0,2,0,1,0,0,0,0]
5
A[3]=8 → C[8]++ → C = [0,0,2,0,1,0,0,0,1]
Phase 2: PREFIX SUMS — O(k) time
For i=1 to k: C[i] += C[i-1]. After this, C[i] = number of elements with value ≤ i. This converts a "histogram" into "output positions." No decisions, no comparisons — pure arithmetic.
1
Start: C = [0, 0, 2, 0, 1, 0, 0, 0, 1]
2
i=1: C[1] += C[0] → 0+0 = 0
3
i=2: C[2] += C[1] → 2+0 = 2
4
i=3: C[3] += C[2] → 0+2 = 2
5
i=4: C[4] += C[3] → 1+2 = 3 ← 3 elements ≤ 4
✓
Result: C = [0, 0, 2, 2, 3, 3, 3, 3, 4]
Phase 3: PLACE Right-to-Left — O(n) time
Scan A from right→left. For each A[j]: write to B[C[A[j]]−1], then decrement C[A[j]]. The right-to-left direction ensures stability. No comparison is ever made — just index arithmetic.
1
j=3, A[3]=8 → B[C[8]−1] = B[3] = 8. C[8]=3
2
j=2, A[2]=2 → B[C[2]−1] = B[1] = 2. C[2]=1
3
j=1, A[1]=2 → B[C[2]−1] = B[0] = 2. C[2]=0
4
j=0, A[0]=4 → B[C[4]−1] = B[2] = 4. C[4]=2
✓
B = [2, 2, 4, 8] — Sorted! Zero comparisons total.
✅ What Paradigm They Actually Use: Integer Structure Exploitation
The Core Insight
Counting and Radix Sort belong to non-comparison-based integer sorting. They do not fit D&C, DP, or Greedy because their design principle is different: they exploit the algebraic properties of integers to place elements directly without comparisons.
How They Exploit Integer Structure
• Integers as array indices — if A[i]=7, directly access C[7]. Floats/strings cannot do this.
• Digit extraction — getDigit(n, pos) in O(1), enabling Radix Sort's digit passes.
• Bounded enumerable range — preallocate exactly C[0..k] knowing all possible values.
Why This Beats Ω(n log n)
The Ω(n log n) lower bound applies only to algorithms that determine order exclusively through comparisons. By using integer values as indices, Counting Sort never compares A[i] to A[j] — it reads A[i] and uses its value to address C[A[i]], completely sidestepping the decision-tree argument.
The Algorithm Pattern
The pattern is: Count → Accumulate → Place. This three-phase structure is unique to integer sorting. No recursion tree, no DP table, no greedy choice — just direct positional arithmetic from integer properties.

If You Tried D&C (like Merge Sort) on Integer Data
Merge Sort works correctly on integers but pays O(n log n) comparisons when Counting Sort sorts in O(n+k) with none. For n=10,000 elements in range 0–1000: Merge Sort needs ~130,000 comparisons, Counting Sort needs only ~11,000 ops.
