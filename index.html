Asymptotic Notation — The Five Forms

When we write T(n) = O(f(n)) we're describing how the running time grows as n → ∞. There are five standard forms, each making a different claim about the relationship between T(n) and f(n). Mastering all five is essential for CLO-1.

📐 Exact Runtime Function — Where the Notation Comes From

Before applying any asymptotic notation, we need the exact function T(n). Counting every loop iteration and array access gives the following — this is the raw f(n,k) from which all five notations are derived:

Counting Sort — Exact f(n, k)
f(n, k) = 3n + 2k + 5
Phase 1: n reads + n writes = 2n ops
Phase 2: k reads + k writes = 2k ops
Phase 3: n reads + n writes + n reads = 3n ops (+5 constants)
→ Θ(n + k)
Radix Sort — Exact f(n, d, b)
f(n, d, b) = d · (3n + 2b + 5)
d passes, each = one Counting Sort on b digit values
d = ⌈log_b(max value)⌉, b = base (10 for decimal)
Example: max=999, b=10 → d=3 passes → 3·(3n+25)
→ Θ(d · (n + b))

Why this matters: The five notations below are all derived from this one function. O(n+k) uses c=4; Ω(n+k) uses c=2; Θ(n+k) uses both; o(n²) uses the limit test; ω(log n) uses the limit test. Use the 🧮 Calculator tab to plug in real n and k values and verify each notation numerically.

📊 How the Five Forms Relate to Each Other

Think of them on a scale from "strictly slower" to "strictly faster". Θ is the most precise — it pins the exact growth class. O and Ω are one-sided bounds. o and ω are the strict variants (no equality allowed). Θ = O ∩ Ω — a function is Θ(g) if and only if it is both O(g) and Ω(g). Equivalently: ω ⊂ Ω and o ⊂ O, meaning every strict bound is also a non-strict bound.

▲ UPPER BOUND
O
Big-O Notation
f(n) = O(g(n)) iff ∃ positive constants c and n₀ such that:

0 ≤ f(n) ≤ c · g(n)  for all n ≥ n₀

This is an asymptotic upper bound. f grows no faster than g (within a constant factor). It is the most commonly used notation — it gives the worst-case ceiling.
T(n) = O(n+k)  for Counting Sort
⟹ ∃ c, n₀ : T(n) ≤ c·(n+k) for all n ≥ n₀
⟹ c = 4 works: 3n+2k+5 ≤ 4(n+k) when n,k ≥ 5

T(n) = O(d·(n+b))  for Radix Sort
⟹ d passes × O(n+b) each. b = base, d = digit count
Key rule: O only gives a ceiling. O(n+k) is technically also O(n²) — but O(n+k) is tighter and more useful. Always state the tightest valid Big-O.
Examples: 5n+3 = O(n), n log n = O(n²), n² = O(n³)
▼ LOWER BOUND
Ω
Big-Omega Notation
f(n) = Ω(g(n)) iff ∃ positive constants c and n₀ such that:

0 ≤ c · g(n) ≤ f(n)  for all n ≥ n₀

This is an asymptotic lower bound. f grows at least as fast as g. Used to prove that no algorithm can do better than a certain bound.
T(n) = Ω(n+k)  for Counting Sort
⟹ Must scan all n input elements (Ω(n))
⟹ Must initialize all k+1 count cells (Ω(k))

Comparison sorts: T(n) = Ω(n log n)
⟹ Any comparison sort needs ≥ n·log₂n comparisons (decision tree)
The famous Ω(n log n) lower bound means NO comparison-based sort can ever beat n log n — it's a mathematical impossibility, not an engineering limitation.
Examples: n³ = Ω(n²), n log n = Ω(n), n+k = Ω(k) when k ≥ n
◆ TIGHT BOUND
Θ
Theta Notation
f(n) = Θ(g(n)) iff both f(n) = O(g(n)) and f(n) = Ω(g(n)).

Equivalently: ∃ c₁, c₂, n₀ > 0 such that:

c₁·g(n) ≤ f(n) ≤ c₂·g(n)  for all n ≥ n₀

This is the tightest bound — f and g grow at exactly the same rate asymptotically.
T(n) = Θ(n+k)  for Counting Sort
⟹ Best = Avg = Worst = O(n+k) = Ω(n+k)
⟹ Θ applies when all cases share the same growth

T(n) = Θ(d·(n+b))  for Radix Sort
⟹ Every case does exactly d passes of O(n+b)
When to use Θ: when best case = worst case (no variance). Both Counting Sort and Radix Sort have this property — they always do the same number of passes.
Examples: n²/2 + 3n = Θ(n²), 7n log n = Θ(n log n)
▲ STRICT UPPER BOUND
o
Small-o Notation
f(n) = o(g(n)) iff for every positive constant c, ∃ n₀ such that:

f(n) < c · g(n)  for all n ≥ n₀

Equivalently:  limn→∞ f(n)/g(n) = 0

f grows strictly slower than g — the ratio collapses to zero. Unlike Big-O, equality is not allowed.
Counting Sort T(n) = O(n+k) when k = O(n):

n+k = o(n²)
⟹ lim (n+k)/n² = lim (2n)/n² = lim 2/n = 0 ✓

n+k = o(n·log²n)  when k = O(n)
⟹ lim n/(n·log²n) = lim 1/log²n = 0 ✓
Difference from Big-O: Big-O allows f(n) = c·g(n). Small-o requires the ratio to go to zero — a strictly loose bound.
Examples: n = o(n²), n log n = o(n²), but 2n ≠ o(n)
▼ STRICT LOWER BOUND
ω
Small-omega Notation
f(n) = ω(g(n)) iff for every positive constant c, ∃ n₀ such that:

f(n) > c · g(n)  for all n ≥ n₀

Equivalently:  limn→∞ f(n)/g(n) = ∞

f grows strictly faster than g — the ratio explodes to infinity. Unlike Big-Ω, equality is not allowed.
n+k = ω(log n)  when k = O(n)
⟹ lim n/log n = ∞ ✓ (linear beats log)

n² = ω(n log n)
⟹ lim n²/(n log n) = lim n/log n = ∞ ✓

So: O(n²) sorts are ω(n log n) — they grow infinitely faster
Difference from Big-Ω: Big-Ω allows equality (f could equal c·g). Small-ω requires the ratio to be unbounded — no constant c can hold it down.
Examples: n² = ω(n log n), n = ω(log n), n! = ω(2ⁿ)

Decision Tree — Proof of Ω(n log n) Lower Bound

Every comparison-based sorting algorithm can be modeled as a binary decision tree. Each internal node is a comparison A[i] ≤ A[j]. Each leaf is a possible sorted permutation. The depth of the tree equals the worst-case number of comparisons. This gives us the mathematical proof.

Decision Tree for 3 Elements [a, b, c]

For n=3 elements: 3! = 6 possible sorted permutations, so the tree needs at least 6 reachable leaves. A binary tree of depth h has at most 2ʰ leaves, so 2ʰ ≥ 6 → h ≥ log₂(6) ≈ 2.58 → at least 3 comparisons needed. For general n: h ≥ log₂(n!) = Θ(n log n) by Stirling's approximation. Dashed branches (∅) are unreachable due to transitivity — shown for tree symmetry only. Click any node to highlight its path.

Why Counting & Radix Sort Bypass This Bound

The Ω(n log n) proof applies only to algorithms that determine order exclusively through comparisons. The decision-tree model counts how many A[i] ≤ A[j] questions are asked. If you never ask them — the proof doesn't apply.

Counting Sort's Key Operation
C[A[i]] += 1  → no A[i] ≤ A[j] !
A[i] is used as an array address, not compared to A[j]. This is only possible because A[i] is an integer — integers can be array indices directly.
Three Integer Properties Exploited
1. Integers as indices: A[i]=7 → access C[7] in O(1)
2. Digit extraction: getDigit(n, pos) in O(1)
3. Bounded range: preallocate exactly C[0..k]
Conclusion
By using integer values as indices instead of comparing them, Counting Sort completely sidesteps the decision-tree argument. Information about element order is extracted from the values themselves via O(1) array indexing — not from pairwise comparison. Result: O(n+k) instead of Ω(n log n).

Step-by-Step Proof — Ω(n log n) Lower Bound for Comparison Sorts

1
n! possible sorted outputs
For n distinct elements, there are n! permutations. Any correct sorting algorithm must be able to produce each one — so the decision tree needs at least n! reachable leaves.
2
Binary tree depth ≥ log₂(n!)
A binary tree of depth h has at most 2ʰ leaves. We need ≥ n! leaves, so 2ʰ ≥ n!, which gives us h ≥ log₂(n!).
3
Stirling: log₂(n!) = Θ(n log n)
By Stirling's approximation: n! ≥ (n/e)ⁿ, so log₂(n!) ≥ n·log₂(n/e) = n·log₂n − n·log₂e = Θ(n log n). Therefore depth h = Ω(n log n). QED.

Algorithm Overview

Two linear-time sorting algorithms that exploit integer structure to break the Ω(n log n) comparison-sort lower bound. This section covers how they work, their history, and when to use them.

Counting Sort
Sorts integers in range [0, k] by counting occurrences, then using prefix sums to place each element at its exact output position.
T(n) = Θ(n + k) Space = O(n + k) Stable ✓
1
Count: Build array C[0..k]. For each element A[j], increment C[A[j]]++. C[i] now holds how many times value i appears.
2
Prefix Sum: For i = 1 to k: C[i] += C[i-1]. Now C[i] holds the number of elements ≤ i — meaning the last output position for value i.
3
Place (reverse): Traverse A right-to-left. For each A[j]: set B[C[A[j]]-1] = A[j], then decrement C[A[j]]--. This ensures stability.
Radix Sort
Sorts d-digit integers digit-by-digit from least significant to most significant, using a stable sub-sort (Counting Sort) at each digit pass.
T(n) = Θ(d · (n+b)) Space = O(n + b) Stable ✓
1
Find d: Count the number of digits in max(A). Each digit pass uses base b (typically 10). d = ⌈logb(max value)⌉.
2
Digit Passes: For each digit position p = 0 (ones) to d-1 (most significant): extract the digit using (A[i] / bp) % b, then run stable Counting Sort on those digit values.
3
Why LSD (Least Significant First): Stability of each pass guarantees that elements equal in the current digit maintain their relative order from previous passes. MSD-first would require recursion or careful bookkeeping.
⚡ Why These Algorithms Run in Linear Time

The key insight: these algorithms never ask "is A[i] ≤ A[j]?" — that's what comparison-based algorithms do, and it's what the Ω(n log n) lower bound is about. Instead:

Integer as Index
Since values are integers in [0,k], we use A[i] directly as an array index: C[A[i]]++. This is O(1) — no comparison, no sorting, just memory access.
Bounded Range = Bounded Work
We know the range is [0,k], so we preallocate exactly k+1 buckets. When k = O(n), processing all buckets costs O(n) — not O(n log n).
Stability Transfers Order
Radix Sort's LSD pass order works because stable sub-sorts preserve prior digit rankings. Without stability, a later pass would destroy the order established by earlier passes.
🕰 Brief History
1887 — Hollerith Tabulating Machine
Herman Hollerith invented electromechanical punched-card machines for the US Census. They sorted cards by physically routing them into bins based on hole positions — the conceptual origin of radix/bucket sorting.
1954 — Harold Seward
Harold H. Seward formally described Counting Sort in an MIT master's thesis, giving it a rigorous algorithmic treatment. Also credited for early work on Radix Sort in the digital computing context.
1960s — IBM Card Sorters
LSD Radix Sort became the algorithm behind physical card-sorting machines used in businesses and government. Cards were passed through machines digit-by-digit, sorting enormous datasets with no comparisons.
2001 — CLRS Codifies the Theory
Cormen, Leiserson, Rivest & Stein's Introduction to Algorithms (Chapter 8: Sorting in Linear Time) formalizes Counting Sort, Radix Sort, and Bucket Sort with complete proofs and complexity analysis. This is the reference for this course.
✅ When to Use These Algorithms
k = O(n) — Range Proportional to Input
Example: n students, scores in [0,100]. k=100 is constant, so T(n) = Θ(n+100) = Θ(n). Truly linear.
Non-negative integers with a known bounded range
Counting Sort requires knowing max(A) in advance to preallocate C[0..k]. Values must be integers since they are used as array indices.
Stability Required
Both algorithms are stable. Radix Sort depends on stability — using an unstable sub-sort breaks correctness entirely. Ideal when sort key is part of a larger record.
Zero variance in performance
Θ(n+k) always — no best/worst case distinction, no pivot bad luck. Perfectly predictable runtime, ideal for real-time systems.
⛔ When NOT to Use These Algorithms
Non-integer data
Floats, strings, objects, UUIDs cannot be used as array indices. Counting Sort is physically impossible. Use Merge Sort or Quicksort instead.
k ≫ n — Huge Value Range
Input [1, 9999] for n=2 elements: count array needs 10,000 cells. O(k)=10,000 vs O(n log n)=2. Comparison sort wins decisively.
Memory-constrained environments
Counting Sort always allocates O(n+k) extra memory. If memory is tight, in-place Quicksort (O(log n) stack) is far better.
Unknown or dynamic range
If you don't know max(A) upfront, you can't preallocate C. Comparison sorts work on any data without prior range knowledge.

Why Does LSD Work? — The Stability Argument
After Pass 1 (ones), elements with equal ones digits are sorted by nothing prior. After Pass 2 (tens), elements with equal tens digits maintain their Pass 1 order (stable sort!), so they are correctly ordered by ones digit within the same tens group. After Pass 3 (hundreds), the same logic applies. Each pass refines, never breaks, prior work. If the sub-sort were unstable, Pass 2 would scramble the order established by Pass 1 — the algorithm would be incorrect.

🔐 Stability Deep Dive — Why the Reverse Traversal Makes Counting Sort Stable

Consider two records with the same key value 3: record X at input position 2 and record Y at input position 6. X appears before Y in the input, so a stable sort must keep X before Y in the output. After prefix sums, C[3] = 6, meaning the last 1-indexed output slot for value 3 is position 6 (i.e. B[5] in 0-indexed). Forward traversal (wrong): j=2 → X placed at B[5], j=6 → Y placed at B[4]. Result: Y before X — stability broken. Reverse traversal (correct): j=6 → Y placed at B[5], j=2 → X placed at B[4]. Result: X at B[4] before Y at B[5] — original order preserved. ✓

Forward traversal (WRONG — unstable)
j=2: X has value 3, C[3]=6, place at B[5]. C[3] becomes 5.
j=6: Y has value 3, C[3]=5, place at B[4]. C[3] becomes 4.
Result: Y at B[4] before X at B[5] — order reversed!
Reverse traversal (CORRECT — stable)
j=6: Y has value 3, C[3]=6, place at B[5]. C[3] becomes 5.
j=2: X has value 3, C[3]=5, place at B[4]. C[3] becomes 4.
Result: X at B[4] before Y at B[5] — original order preserved!

CLO-2 — Sorting & Searching Role

CLO-2 asks: how is sorting used inside these algorithms, what happens when you change or remove the sorting step, and how do they compare to a classical comparison sort on the same input? This section covers all three requirements. The interactive visualiser on the ⚡ Visualizer page has the full playback controls (Build Steps / Step / Play / Pause / Reset) and live history table.

🔢 How Sorting Is Used Inside Counting Sort

Counting Sort does not sort by comparing elements. Instead it uses the values themselves as array indices. The three phases each play a distinct, essential role:

1
Phase 1 (Count): Scans the input to count how many times each value appears. This is the information-gathering step — it replaces the comparison step of classical sorts. Code: C[A[j]]++. Cost: Θ(n).
2
Phase 2 (Prefix Sum): Converts raw counts into output positions. This is the ordering step — it determines where each value goes without any pairwise comparison. Code: C[i] += C[i-1]. Cost: Θ(k).
3
Phase 3 (Place): Uses the computed positions to move every element into its correct location in one pass. The right-to-left scan is mandatory for stability. Code: B[C[A[j]]-1] = A[j]. Cost: Θ(n).
The Key Insight — Sorting Role
Phase 2 (Prefix Sum) is the heart of the sorting mechanism. Without it, every element would try to write to position 0. Phase 2 transforms a count histogram into a position map — this is what achieves sorted order without ever asking "is A[i] ≤ A[j]?"
🔬 Impact Analysis — What Breaks When Each Phase Is Removed or Changed
❌ Remove Phase 2 (Prefix Sum)
Input A = [3, 1, 2] → C after counting: [0, 1, 1, 1]
B[C[2]-1] = B[-1] → out of bounds crash
or all elements pile at index 0
Why: C[i] holds a raw count (how many times i appears), not a position. Phase 2 is what converts "count = 1" into "position = 3." Without it there is no mapping from value to output slot — the algorithm is logically broken and will produce wrong output or crash.
❌ Change Phase 3 to Forward Traversal (left → right)
Input A = [3★, 1, 3●, 2] (two records with value 3)
Forward: 3★ → B[5], 3● → B[4]
Result: 3● before 3★ — reversed!
Why: Stability is broken. Equal-key elements no longer keep their original relative order. This makes Radix Sort produce incorrect results because each digit pass relies on the previous pass's stable ordering being preserved.
❌ Use Unstable Sub-Sort Inside Radix Sort (e.g. Quicksort)
A = [329, 457, 839] after ones pass: [329, 839, 457]
Unstable tens pass: 329 and 839
may swap → final order wrong
Why: LSD Radix Sort only works because each digit pass is stable — it refines without destroying prior passes' work. An unstable sub-sort scrambles equal-digit elements, making the algorithm produce completely incorrect output — not just slower, but wrong.

Why the "Worst Case" Is Different for Counting Sort
For comparison sorts (Quicksort, Merge Sort), worst case is about input order — e.g. a sorted or reverse-sorted array for Quicksort with bad pivot choice. For Counting Sort, input order is irrelevant — it always costs Θ(n+k) regardless of order. The worst case is purely about the range k being large relative to n. This is why Counting Sort is described as having Θ(n+k) in all cases — there is no best/worst input order distinction.

Design Paradigm — CLO-3

Counting Sort and Radix Sort are not Divide & Conquer, Dynamic Programming, or Greedy. They belong to a distinct paradigm: Non-Comparison / Integer Exploitation. Understanding this difference is core to CLO-3 (10 marks).

⚡ The Core Claim You Must Be Able to Defend
Counting Sort and Radix Sort are "Non-comparison sorting algorithms that exploit the integer structure of input data". They do not divide, do not memoize, do not make locally optimal choices. They extract information from values as memory addresses, not from pairwise comparisons or recursive substructure. This is why they break the Ω(n log n) lower bound — and why the comparison-sort paradigms cannot.
🌳 Merge Sort — Divide & Conquer Recursion Tree

For A = [4, 2, 8, 3]: the array is recursively split into halves, each half is sorted independently, then merged. The tree shows the recursive call structure.

Each node = a recursive sub-call on a subset of data. Each internal comparison (A[i] ≤ A[j]) reduces the set of possible orderings. Depth = log₂n, giving Θ(n log n) total comparisons. The Ω(n log n) lower bound applies to this model.

⚡ Counting Sort — Sequential Three-Phase Scan (NOT D&C)

For A = [4, 2, 8, 3]: no splitting, no recursion, no merging. Three sequential left-to-right (or right-to-left) scans over the entire array.

No recursive calls. No tree. No merging of subproblems. The entire array is processed three times sequentially. There is no A[i] ≤ A[j] comparison anywhere. The decision-tree lower bound proof simply has nothing to count — so Ω(n log n) does not apply.

Divide & Conquer
✗ NOT this
Split problem into independent subproblems → solve recursively → combine.

Examples: Merge Sort (split n→n/2→…), Quicksort, Binary Search, FFT.

Why counting sort isn't D&C: It never splits the array into recursive subproblems. There is no recursion tree. There are no independent sub-sorts being combined. It's a single sequential three-phase scan.
Dynamic Programming
✗ NOT this
Solve overlapping subproblems once, store results → build up solution bottom-up or top-down.

Examples: LCS, Knapsack, Bellman-Ford, Matrix Chain Mult.

Why counting sort isn't DP: There are no overlapping subproblems. The prefix sum step looks like DP (C[i] = C[i] + C[i-1]) but it's just a prefix-sum scan — there is no optimal substructure being exploited.
Greedy
✗ NOT this
At each step, make the locally optimal choice → leads to globally optimal solution.

Examples: Dijkstra's, Prim's, Huffman Coding, Activity Selection.

Why counting sort isn't Greedy: There are no "choices" made. Every element goes to exactly one predetermined position. Nothing is selected or skipped.
Non-Comparison Sorting
(The Real Paradigm)
✓ THIS ONE
Exploit domain knowledge about input (values are integers in [0,k]) to extract order via O(1) array indexing rather than pairwise comparisons.

Key mechanism: C[A[i]]++ — using the value as an address. No A[i] ≤ A[j] ever asked.

Consequence: Decision-tree lower bound doesn't apply → Θ(n+k) instead of Ω(n log n).
Why D&C Sorts Can't Beat Ω(n log n)
Key Contrast
Merge Sort, Quicksort, Heap Sort — all use A[i] ≤ A[j] to determine order.

Every comparison eliminates one possible permutation. There are n! permutations. A binary decision tree of depth h can eliminate at most 2ʰ permutations.

So 2ʰ ≥ n! → h ≥ log₂(n!) = Θ(n log n). No comparison sort can escape this.
🧠 Tricky Question: Radix Sort Uses Counting Sort as a Sub-Routine — Is That D&C?

A subtle point that often comes up: Radix Sort calls Counting Sort d times. Isn't calling a sub-routine "divide" the problem? No — here's why:

What D&C Requires (Not Present Here)
Divide & Conquer requires splitting the same problem instance into independent sub-problems of the same type, solving them separately, and combining results. Each sub-problem in D&C works on a subset of the original data and produces a partial solution that is merged.
What Radix Sort Actually Does
Radix Sort applies the same operation (sort by digit p) to the entire array d times. There is no splitting, no independence, no merging of partial results. Calling a helper function on the full array is iteration, not recursion or divide-and-conquer. The paradigm is sequential integer exploitation, not D&C.

⚠️ Counterexample — Where the Wrong Paradigm Breaks (CLO-3 Requirement)

The project requires showing an example where a different paradigm gives a wrong or worse answer. Here are two:

Scenario 1: Huge Range, Small n
Input: [1, 1000000] (n=2 elements, k=10⁶). Counting Sort: O(k) = O(10⁶) — allocates a million-cell array for 2 elements. Merge Sort: O(n log n) = O(2) — one comparison. D&C paradigm wins by 500,000×.
Scenario 2: Float Data
Input: [3.7, 1.2, 4.9]. Counting Sort is physically impossible — you cannot use 3.7 as an array index. Any comparison-based sort works fine. The integer-exploit paradigm simply doesn't apply.
Scenario 3: If You Used Unstable Sub-Sort in Radix Sort
Use Quicksort (unstable) as the sub-sort in Radix Sort. After Pass 1 (ones): [720, 355, 436, 457, 657, 329, 839].
Pass 2 (tens) uses Quicksort — it does not preserve the ones-digit order for ties. Elements with equal tens digits may be reordered arbitrarily, producing an incorrect final result.
Scenario 4: Trying DP on Sorting
DP requires overlapping subproblems with optimal substructure. Sorting a subarray does not help sort a different subarray — there's no state reuse. Applying DP here would just be Merge Sort recursion with a useless memo table, giving worse constants for no benefit.

🚫 Common Mistakes — Learn These Before the Demo
❌
Mistake 1: "Counting Sort is O(n)" — Wrong. It's O(n + k). When k ≫ n (e.g. sorting 2 elements from [0, 10⁶]), it's O(k) = O(10⁶), far worse than O(n log n) = O(2).
❌
Mistake 2: Thinking forward traversal in Phase 3 is fine — It breaks stability! The reverse (right-to-left) traversal is required for stability, which is essential when Counting Sort is used as Radix Sort's sub-routine.
❌
Mistake 3: "Radix Sort is O(n log n)" — No. Each digit pass costs Θ(n+b). With d passes, total = Θ(d·(n+b)). If d and b are constants (e.g. base 10, fixed integer size), this is Θ(n) — linear, not logarithmic.
❌
Mistake 4: "Counting Sort is D&C because it has sub-steps" — D&C requires recursive decomposition into independent subproblems. Counting Sort's three phases are sequential scans of the entire array, not independent recursive sub-computations.
❌
Mistake 5: Using an unstable sort inside Radix Sort — If you replace Counting Sort with Quicksort (unstable) inside Radix Sort, the algorithm produces incorrect results, not just slower ones. Stability is a correctness requirement, not a performance preference.
❌
Mistake 6: Claiming Θ notation means "average case" — Θ means the exact asymptotic growth class (tight bound from both above and below). Counting Sort is Θ(n+k) in all cases — there is no best/worst distinction. This is different from Quicksort which has Θ(n²) worst case.
