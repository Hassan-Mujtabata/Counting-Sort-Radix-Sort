Welcome to Non-Comparison Sorting

This tool visualizes Counting Sort and Radix Sort. These algorithms achieve linear sorting time by exploiting the properties of integers, bypassing the mathematical $\Omega(n \log n)$ lower bound that restricts standard comparison sorts (like Merge Sort or Quicksort).

How Counting Sort Works

Instead of comparing elements (`A[i] < A[j]`), it uses the element's value directly as an array index.

  1. Count: Create an array `C` of size `k+1` (where k is max value). Tally occurrences: `C[A[i]]++`.
  2. Prefix Sums: Modify `C` so `C[i]` contains the number of elements $\le i$. This determines exact output positions.
  3. Place (Right-to-Left): Iterate input backwards to ensure stability. Place `A[j]` at output index `C[A[j]]-1` and decrement the count.
How Radix Sort Works

Radix Sort handles larger numbers by sorting them digit-by-digit, from Least Significant Digit (LSD) to Most Significant Digit (MSD).

  • It uses Counting Sort as a stable subroutine for each digit pass.
  • Stability is vital: when sorting the "tens" digit, numbers with the same "tens" digit must remain sorted by their "ones" digit from the previous pass.
  • Number of passes $d$ equals the maximum number of digits.

CLO-1: Complexity Analysis & Formal Asymptotics

To mathematically analyze algorithms, we use five formal asymptotic notations to describe bounds on running time $f(n)$ relative to a simple function $g(n)$ as $n \to \infty$.

O
Big O (Upper Bound)
f(n) ≤ c · g(n)

Algorithm will never run slower than this. Worst-case scenario.

Ω
Big Omega (Lower Bound)
f(n) ≥ c · g(n)

Algorithm will take at least this much time. Best-case scenario.

Θ
Big Theta (Tight Bound)
c₁·g(n) ≤ f(n) ≤ c₂·g(n)

Both upper and lower bounded by the same function family.

o
Little o (Strict Upper)
f(n) < c · g(n)

Grows strictly slower than g(n).

ω
Little omega (Strict Lower)
f(n) > c · g(n)

Grows strictly faster than g(n).

CLO-2: Role of Sorting & Impact Analysis

The Role of the Sub-Sort in Radix

In Radix Sort, sorting is applied as a pipeline of passes. Each pass sorts the array by exactly one digit (ones, tens, hundreds). Counting Sort is chosen as the engine for this sub-sort because it is O(n) and, crucially, it is stable.

Input: [170, 45, 75]

Pass 1 (Ones): C-Sort(0, 5, 5) → [170, 45, 75]
Pass 2 (Tens): C-Sort(7, 4, 7) → [45, 170, 75]
Pass 3 (Hund): C-Sort(1, 0, 0) → [45, 75, 170]
Impact of Removing Stability

If we used an unstable sort (like standard Quicksort) for the digit passes, equal digits would be scrambled, destroying the work of previous passes.

CLO-3: Design Paradigm (Integer Structure Exploitation)

Counting Sort does not fall into the standard paradigms of Divide & Conquer, Dynamic Programming, or Greedy algorithms. It uses a different paradigm entirely: Integer Index Exploitation.

✗ Not Divide & Conquer

D&C splits the array recursively into sub-problems (like Merge Sort). Counting Sort never splits the array. There is no recursion tree, and no 'Merge' step. It processes the array in 3 flat, linear loops.

✗ Not Dynamic Programming

DP relies on overlapping subproblems and memoization. While Counting Sort uses a secondary array `C`, it is not memoizing a recurrence relation. It is simply building a histogram.

✗ Not Greedy

Greedy algorithms make local optimum choices at each step. Counting Sort makes no choices. Placement is entirely deterministic arithmetic computed from prefix sums.

✓ Integer Structure Exploitation

This paradigm exploits the fact that integers can act as RAM addresses in O(1) time. By using the value of `A[i]` directly as an index for `C`, it completely bypasses the need to compare `A[i]` against `A[j]`.
